# クレデンシャルの自己増殖

## 自己増殖とは

key-rest は LLM エージェントがクレデンシャルの値を**見る**ことを防ぎますが、エージェントがクレデンシャルを**使って** API 呼び出しにより同種の新しいクレデンシャルを生成することは防げません。サービスの API が既存のクレデンシャルを使って同種のクレデンシャルを作成できる場合、エージェントは以下のことが可能です：

1. key-rest 経由でクレデンシャル作成エンドポイントを呼び出す
2. レスポンスボディに含まれる新しいクレデンシャルを受け取る
3. 新しいクレデンシャルは key-rest のクレデンシャルストアに**ない**ため、レスポンスマスキングで検出されない

これはクレデンシャルプロキシモデルのアーキテクチャ上の制約です（[issue #9](https://github.com/koteitan/key-rest/issues/9)）。この性質を**自己増殖**と呼びます — クレデンシャルが同種のクレデンシャルを作成できるかどうか（key-X → key-X）を指します。

## サービスごとの調査

### GitHub

| クレデンシャル | 自己増殖？ | 詳細 |
|---|---|---|
| PAT（classic/fine-grained） | **不可** | PAT を作成する API エンドポイントが存在しない。`POST /authorizations` は 2020 年に廃止。 |
| OAuth トークン | **不可** | 新しいトークンの発行には OAuth フロー（ブラウザ認可 + `client_id` + `client_secret`）が必要。いずれも OAuth トークン自体からは取得できない。 |
